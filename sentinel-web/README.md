# Sentinel Web

**Sentinel Web** is the client-facing web application for **Ruckshuck Sentinel**, an AI-assisted Security Operations Center (SOC) platform.

This application provides authenticated users with dashboards and workflows to configure security data sources, run investigations, and view results produced by Sentinel backend services.

---

## Purpose

Sentinel Web is responsible for:

- User authentication flows and session-aware UI (login, logout, protected routes)
- Providing a secure, user-specific dashboard
- Collecting platform configuration input (e.g., Google service account details)
- Triggering backend jobs (via Sentinel APIs)
- Displaying security-relevant data (e.g. access context, geolocation, and VPN indicators) retrieved from backend services
- Surfacing notifications and task results generated by SOC services

Sentinel Web **does not** perform security analysis itself. All data processing and enrichment happens in backend services.

---

## Technology Stack

Sentinel Web is intentionally lightweight and conservative in its technology choices to minimize attack surface.

- **Framework:** Astro
- **Styling:** Tailwind CSS
- **Rendering Model:** Static pages + client-side API calls
- **Authentication (MVP):** Implemented within Sentinel Web, with credentials verified by the backend API and sessions established using secure HttpOnly cookies or tokens
- **State Management:** Minimal client-side state only where required
- **Deployment Target:** CDN-backed static hosting (e.g., Netlify, Cloud CDN, or S3 + CloudFront)

Astro is used to keep JavaScript payloads small and to avoid shipping a heavy single-page application unless it becomes necessary.

---

## Repository Scope

This repository contains **only the web application**.

It does **not** contain:
- Backend APIs
- Job execution logic
- Credential storage
- Security detection or response logic

Those responsibilities live in separate Sentinel services.

## Integration with Other Sentinel Services

Sentinel Web communicates with backend services over HTTPS APIs.

### Primary Integration (MVP)

- **`sentinel-access-context-api`**
  - Fetches Google Workspace login events via the Admin Reports API
  - Enriches login IPs with geolocation, VPN, proxy, and ASN context
  - Returns table-ready JSON for display in the UI

Typical flow:
1. User logs into Sentinel Web
2. User enters platform configuration (e.g., Google service account JSON + date range)
3. Sentinel Web sends a request to `sentinel-access-context-api`
4. Backend processes the request and returns enriched access context data
5. Sentinel Web renders results in a dashboard table

### Future Integrations

Sentinel Web is designed to later integrate with:
- Authentication and RBAC services
- Detection and alerting services
- Case management services
- AI advisory agent services
- Additional platform connectors (cloud, IdP, SaaS)

---

## Security Model (Important)

- Sentinel Web **never stores** platform credentials long-term
- Sensitive credentials are transmitted securely to backend APIs and used in-memory only
- All API communication is over HTTPS
- Authentication tokens are managed by backend services (preferably via HttpOnly cookies)
- Sentinel Web does not log or persist secrets in the browser

This separation is intentional and critical for operating in fintech and crypto environments.

## Local Development

### Prerequisites

- Node.js 18+ (LTS recommended)
- npm or pnpm

### Setup

1. **Clone and install:**
   ```bash
   git clone <repo-url>
   cd sentinel-web
   npm install
   ```

2. **Environment variables:**
   ```bash
   cp .env.example .env
   ```
   Default values in `.env` are suitable for local development. For production, update `SESSION_SECRET` and set `NODE_ENV=production`.

### Run the App Locally

```bash
npm run dev
```

The application will be available at `http://localhost:3000`.

### Local Development Workflow

1. **Sign up:** Navigate to `/signup` and create a test account.
2. **Login:** Use your credentials to log in on `/login`.
3. **Dashboard:** Once authenticated, you'll see the dashboard at `/dashboard`.
4. **Test API Integration:** Click "Fetch Sample Data" to call `http://localhost:8000/api/sample-data` from the sentinel-access-context-api service.

### Notes on Local Testing

- **Authentication:** Sessions are stored in-memory (MVP). Restarting the server will clear all sessions.
- **CORS:** Ensure sentinel-access-context-api (running on localhost:8000) allows requests from `http://localhost:3000`.
- **Secure cookies:** In development, cookies are not marked `Secure` (to work over HTTP). In production, HTTPS is required.

## Authentication Details

### Session Management

- **Method:** HttpOnly session cookies
- **Session Store:** In-memory (MVP; upgrade to Redis for production)
- **TTL:** 1 hour
- **Endpoints:**
  - `POST /api/signup` — Create account and establish session
  - `POST /api/login` — Authenticate and establish session
  - `GET /api/me` — Check current session validity
  - `POST /api/logout` — Destroy session

### Protected Routes

- `/dashboard` and any future protected routes check the session cookie and call `GET /api/me` on the server side.
- If authentication fails, users are redirected to `/login`.

### Password Security

- Passwords are hashed with PBKDF2 (100,000 iterations).
- Minimum password length: 8 characters.
